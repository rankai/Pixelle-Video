    PUB-4 发布中心与生产链路整合 Entry（2026-07-20）¶

    状态：entry_passed_with_boundary；当前唯一入口已放行至 PUB-INTEGRATION/PUB-4 implementation batch 1，PG-I 已以 passed_with_boundary 放行。

    上位约束¶

    • 来源方案：2026-07-18-desktop-auto-publishing-refactor-implementation-plan.md 阶段 PUB-4/门禁 PUB-E。
    • 上位 Gate：PG-J；前置 PG-I 已通过，但 PUB-4 Entry 未通过前不得修改统一发布编排。
    • 独立审查线程：/root/pg_a_closure_reviewer_v3，只读审查，不直接修改代码。

    本 Entry 冻结目标¶

    1. /api/publish/v2/**、PublishPackage、PublishRun、AccountProfile 是发布中心唯一事实源；/#/publish 是唯一发布页面。
    2. StudioApp 第 6 步只展示 package 摘要、复制/下载和进入发布中心，不再拥有第二套发布编排。
    3. 发布中心统一覆盖“准备发布 / 发布账号 / 发布记录”：账号选择、package 预检、run timeline、字段清单、人工接管和安全状态投影。
    4. 应用中心的数字人口播/图文 handoff 能跳转并恢复同一 package/run，不重复创建 package/ref/run。
    5. 生产 package invalidation、复制/下载安全回退、adapter 失败后的本地交付路径保持可用。
    6. 键盘操作、窄窗口和刷新/离开页面恢复均有可执行测试计划。

    允许范围（Entry 通过后）¶

    • desktop/src/features/publishing/**、desktop/src/StudioApp.tsx 的最小发布中心接线与组件测试。
    • api/routers/publish_v2.py、现有 publish service/repository 的最小集成修复；继续复用既有 PublishPackage/PublishRun/AccountProfile 事实源。
    • 应用中心 handoff 的只读跳转/恢复契约和对应测试、fixture、QA evidence。
    • 发布失败/adapter 不可用时的复制/下载 fallback、run timeline 与字段安全投影。
    • 不新增模型配置源、管理员/RBAC/套餐/支付、多租户或第二浏览器运行时。

    禁止范围与人工暂停点¶

    • 不删除旧 V1/V2 回退路径，不做破坏性数据库迁移，不直接清理旧 profile/session。
    • 不自动点击抖音或其他平台最终发布；不触发真实扫码、第三方授权、验证码、真实上传或账号/内容外部写入。
    • 不在 Entry 阶段实现真实 PUB-4 业务代码；先完成契约、fixtures、失败矩阵和六维审查。
    • 任何需要用户确认、第三方授权或最终发布的动作必须暂停并通知用户。

    PUB-E 验收矩阵（Entry 冻结）¶

    验收项 必须冻结/验证
    首次发布 package → account → preflight → run timeline → waiting_for_human，不得显示 completed/published
    重复发布 相同 package/ref/account 重放复用同一 run；不重复上传/创建 ref
    故障恢复 刷新、离开页面、应用重启后恢复同一 run；adapter 失败不丢 package、项目、资产、复制/下载
    生产失效 ArtifactVersion/content fingerprint 变化使旧 package/ref 失效；旧历史可审计
    UI/无障碍 键盘可达、窄窗口无死路、字段清单与状态语义安全
    兼容回退 关闭发布 V2 或新入口后旧发布/下载/复制仍可用

    Entry 退出条件¶

    • 机器可读 contract/fixture 覆盖上述矩阵与 P0/P1 负例；Entry 定向测试、Ruff、git diff --check 通过。
    • 当前负例覆盖：禁止 published/auto_published 投影、旧 workspace 二次编排、stale package 新 run、账号/包不匹配、fallback secret/绝对路径泄露、缺字段/重复上传、键盘/窄窗死路。
    • 独立审查线程按需求完整性、逻辑正确性、边界情况、代码质量、测试覆盖、实际运行结果六维复验，P0/P1=0。
    • Entry 通过后才允许实现 PUB-4；实现必须逐批提交测试、证据与复审，未 implementation_pass_with_boundary 不得进入 PG-J Gate。

    独立 Entry 复审结果¶

    • 审查线程：/root/pg_a_closure_reviewer_v3，只读审查，未修改代码。
    • 结论：entry_passed_with_boundary，P0/P1=0。
    • 证据：Entry contract/fixture 2 passed；Ruff、git diff --check clean；7 个负例覆盖状态误报、旧编排、stale package、账号/包不匹配、fallback 泄露、字段/重复上传、键盘/窄窗死路。
    • 边界：Entry 不等价 PUB-4 业务实现、真实平台/provider、运行时 fallback/UI/账号/package/run E2E；这些进入 implementation batch 1，继续禁止真实外部动作。
