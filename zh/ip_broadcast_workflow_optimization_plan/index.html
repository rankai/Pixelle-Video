    IP 口播 1-6 工作流优化可执行方案¶

    Summary¶

    本轮优化目标是先把当前已跑通的 IP 口播流程固化为稳定基线，再分阶段提升代码质量、架构边界、性能、安全和 UI 一致性。执行时先把本文档写入 docs/ip_broadcast_workflow_optimization_plan.md，再按阶段实施。

    核心原则：

    • 不一次性大重构。
    • 不改变当前 6 步主流程。
    • 不改变“整段音频 + 整段数字人 + 第 5 步覆盖画面”的生产策略。
    • 先补测试护栏，再统一执行层，再拆复杂模块。
    • 每个阶段独立可验证、可回滚。

    Phase 0：保存当前稳定基线¶

    • 运行：
    • uv run ruff check .
    • uv run pytest -q
    • 确认当前 1-6 主流程可用：
    • 素材来源生成文案
    • AI 改写
    • 声音生成
    • 数字人生成
    • 一键成片
    • 下载/复制发布素材
    • 提交当前未提交改动，作为后续重构基线。
    • 不在本阶段改代码逻辑。

    验收标准：

    • 全量测试通过。
    • 当前改动已提交。
    • 8501 页面可打开，主流程无明显报错。

    Phase 1：补稳定性测试护栏¶

    • 增加端到端级别的流程状态测试：
    • 缺文案时停在第 ½ 步。
    • 有文案时下一步是语音。
    • 有语音但无形象时停在第 4 步。
    • 有形象和语音时可进入数字人。
    • 有数字人视频时可进入一键成片。
    • 有最终视频后进入发布。
    • 增加错误状态测试：
    • TTS 失败时写入第 3 步错误，不清空文案。
    • 数字人失败时写入第 4 步错误，不清空音频。
    • 合成失败时写入第 5 步错误，不清空数字人视频。
    • 任一步失败后 loading 应关闭。
    • 增加素材库测试：
    • 视频素材保存、封面生成、删除。
    • 正在使用的视频素材不能删除。
    • manifest 缺字段时能兼容读取。
    • 增加 RunningHub 配置测试：
    • tts_index_custom.json 必须是 RunningHub wrapper。
    • AI App workflow 必须有 webapp_id 和参数映射。

    验收标准：

    • 新测试先失败后转绿。
    • uv run pytest -q 通过。
    • 不改 UI，不做重构，只补测试和必要小修。

    Phase 2：统一执行层与状态处理¶

    • 新增统一 step runner：
    • run_ipb_step(step_key, pixelle_video)
    • 手动按钮和“一键继续”共用同一套依赖检查、执行、错误写入。
    • 保留现有 run_m2、run_m3、run_m4、run_m5 作为底层执行函数，先不删除。
    • 统一错误处理：
    • 捕获异常。
    • 写入 ipb_step_status[step] = "error"。
    • 写入 step notice。
    • 不清空已有产物。
    • loading 必须关闭。
    • 统一成功处理：
    • 写入产物路径。
    • 更新 step status。
    • 刷新 readiness。
    • 一键继续逻辑只负责循环调用 step runner，不直接处理各模块细节。

    验收标准：

    • 手动点击和一键继续行为一致。
    • 任一步失败后页面不会卡 loading。
    • 既有 1-6 流程测试全部通过。
    • 浏览器验证一次完整默认路径。

    Phase 3：拆分第 5 步模块¶

    将 m5_postproduction.py 拆成更小模块，保持行为不变。

    建议拆分：

    • m5_templates.py
    • 画面模板卡片
    • 模板描述
    • 模板预览
    • m5_video_assets.py
    • 视频素材管理
    • 封面展示
    • 素材选择
    • 删除占用检查
    • m5_overlay_planning.py
    • 段落勾选
    • 创建覆盖组
    • 编辑覆盖组
    • 覆盖组 UI
    • m5_composer.py
    • FFmpeg 合成
    • BGM
    • 字幕
    • 覆盖视频应用
    • m5_publish_assets.py
    • 标题描述标签
    • 封面策略
    • 发布素材摘要

    保持公开入口：

    • render_m5_postproduction(pixelle_video, run_mode)
    • run_m5(pixelle_video)

    验收标准：

    • 拆分后无行为变化。
    • 第 5 步相关测试全部通过。
    • m5_postproduction.py 降到 250 行以内。
    • 新模块职责单一，不互相循环导入。

    Phase 4：上传安全与 manifest 可靠性¶

    • 增加统一文件校验工具：
    • basename 校验，禁止路径分隔符。
    • 扩展名白名单。
    • 文件大小限制。
    • 默认大小限制：
    • 形象图片/封面图片：20MB
    • 参考音频：100MB
    • 视频形象：1GB
    • 覆盖视频素材：1GB
    • manifest 写入改为原子写：
    • 写入 .tmp
    • replace 到正式 manifest
    • manifest 读取兼容：
    • 缺字段自动补默认值。
    • 文件损坏时不崩溃，给 warning 并返回空列表。
    • 删除文件时限制在对应素材库目录内。

    验收标准：

    • 非法扩展名被拒绝。
    • 超大文件被拒绝。
    • manifest 损坏不导致页面崩溃。
    • 删除不会越权删除素材目录外文件。

    Phase 5：拆分第 3 步声音模块¶

    将 m3_voice.py 拆成更小模块，保持 UI 和行为不变。

    建议拆分：

    • m3_tts_config.py
    • local / comfyui 模式配置
    • Edge / Index / Spark 参数
    • m3_voice_references.py
    • 参考音频库
    • 上传/录音保存
    • 参考音频选择
    • m3_preview.py
    • 试听文本
    • 试听音频生成
    • m3_runner.py
    • _build_tts_kwargs
    • run_m3
    • 正式语音生成

    验收标准：

    • m3_voice.py 降到 250 行以内。
    • local Edge、RunningHub Edge、Index、Spark 参数测试全部通过。
    • 参考音频上传/录音/选择不回归。
    • 试听不会覆盖正式音频。

    Phase 6：第一版缓存优化¶

    只做两个缓存，不做 FFmpeg/overlay 缓存。

    • TTS 缓存：
    • key = 文案 hash + TTS 模式 + workflow + 参数 + 参考音频 hash。
    • 命中时复用音频路径。
    • 源文件缺失时缓存失效。
    • 数字人缓存：
    • key = 音频 hash + 形象素材 hash + workflow + prompt + duration。
    • 命中时复用数字人视频路径。
    • 源文件缺失时缓存失效。
    • 缓存目录：
    • data/ip_broadcast_cache/tts/
    • data/ip_broadcast_cache/digital_human/
    • UI 提示：
    • 命中缓存时显示“已复用上次生成结果”。

    验收标准：

    • 相同输入不重复调用 TTS。
    • 相同输入不重复调用数字人。
    • 参数变化会重新生成。
    • 缓存文件缺失时自动重新生成。

    Phase 7：UI 一致性优化¶

    • 主控台增加缺失项提示：
    • 缺文案
    • 缺语音
    • 缺形象
    • 缺数字人视频
    • 缺最终视频
    • 统一 step notice：
    • 每步末尾只显示一个状态结果。
    • 成功、失败、警告样式统一。
    • 统一卡片组件：
    • 画面模板卡
    • 数字人形象卡
    • 视频素材卡
    • 第 5 步固定顺序：
    • 画面模板
    • 视频素材管理
    • 画面规划
    • 高级成片设置
    • 发布素材与封面
    • 画面规划保持当前“勾选连续段落成组”方案，不做拖拽时间轴。
    • 视频素材标签、搜索后续单独做，不进入本轮。

    验收标准：

    • 1280 宽桌面布局无明显高度错乱。
    • 无文案段落时画面规划提示明确。
    • 有分段文案时可创建覆盖组。
    • 卡片标题、描述、按钮高度一致。
    • 长任务失败后 loading 不残留。

    Test Plan¶

    每个阶段完成后运行：

    • uv run ruff check .
    • uv run pytest -q

    关键浏览器回归：

    • 默认链路：
    • 粘贴/提取素材 → AI 改写 → 生成语音 → 生成数字人 → 一键成片 → 下载发布素材
    • IP 学习链路：
    • 主页抓取成功
    • 主页抓取失败后手动粘贴 5 条视频链接
    • 声音链路：
    • local Edge
    • RunningHub Edge
    • RunningHub Index
    • RunningHub Spark
    • 参考音频库选择
    • 数字人链路：
    • 旧 digital_combination workflow
    • 新 AI App workflow
    • 401/失败/超时错误展示
    • 成片链路：
    • BGM 长于视频时最终视频长度按主视频/语音控制
    • 覆盖组上传视频
    • 覆盖组选择视频素材库素材
    • 字幕模板样式生效
    • 素材库：
    • 保存、展示封面、选择、删除
    • 使用中素材删除被阻止

    Assumptions¶

    • 当前已跑通的 6 步工作流不改入口和用户心智。
    • 不恢复独立标题封面步骤，标题封面继续并入第 5 步。
    • 第 5 步覆盖方案继续基于整段音频/整段数字人视频，不拆分生成语音或数字人。
    • 本轮不做视频素材标签、搜索、批量管理。
    • 本轮不接真实发布平台 API。
    • 正式实施前先提交当前稳定基线，避免重构和已有改动混在一起。
