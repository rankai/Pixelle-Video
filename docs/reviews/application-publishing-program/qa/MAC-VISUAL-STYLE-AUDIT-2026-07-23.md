# Mac 页面视觉一致性审查（2026-07-23）

## 总体结论

用户观察成立：发布中心不是完全脱离系统，但目前明显处在“共享壳层 + 独立页面皮肤”的状态。应用中心使用较完整的新设计语言，发布中心 V2 仍保留更简化的旧式内容表达；工作台则是第三套较密集的业务工作流布局。

这属于中等优先级的设计一致性问题，不影响当前发布安全边界，但会降低用户对“这是同一个产品”的感知，也会让发布中心显得像临时接入页面。

## 当前审查步骤

1. 应用中心（1440×900）：页面层级、搜索、分类、四张应用卡片和主按钮。
2. 发布中心/发布账号（1440×900）：V2 标题、人工确认标签、分区 Tab、账号摘要和刷新动作。
3. 口播剪辑工作台（1440×900）：生产状态、步骤条、表单卡片和业务操作。

## 发现

### 1. 发布中心的内容容器层级偏弱

应用中心有明确的 `eyebrow → 大标题 → 说明 → 工具栏 → 卡片网格` 层级；发布中心虽然共享顶部栏和背景，但 V2 内容主要是一段标题、简单 Tab 和一张宽卡片，视觉节奏较空，缺少应用中心的“模块入口感”。

### 2. Tab、标签和按钮表达不统一

应用中心分类使用胶囊按钮和统一的紫色选中态；发布中心使用无边框文字 Tab 和橙色焦点边框，账号状态标签的密度、颜色和间距也与应用卡片不一致。建议统一选中态、标签语义色和按钮层级。

### 3. 发布账号卡片信息密度不足且缺少平台入口感

当前 V2 账号页只展示已有账号分组，视频号暂无账号时完全不出现。对用户而言，平台能力目录是不完整的；应按平台能力显示四个平台，空平台显示“未添加/添加账号”。

### 4. 工作台不必完全变成应用中心卡片风格

工作台的步骤条和表单密度是生产任务所需，可以保留；但其标题、Card、按钮、间距和状态标签应复用同一组设计 tokens，避免“业务页面独立换肤”。

## 建议的统一方向

建议保留应用中心作为视觉基准，发布中心与工作台向它对齐以下共享层，而不强行把所有页面做成同样的卡片网格：

- 页面宽度、左右内边距、顶部标题层级
- `app-bg / surface / border / primary / muted` 颜色 tokens
- Card 圆角、边框、阴影和内部间距
- 主按钮、次按钮、危险按钮的层级
- Tag 的成功、试点、未验证、错误语义色
- Tab 的选中态和键盘焦点态
- 空态、错误态和安全边界 Alert 的结构

优先级建议：

1. 先统一发布中心 V2 的标题/容器/Tab/Tag/Button，不动业务逻辑。
2. 再补齐视频号空平台卡片和账号管理入口。
3. 最后拆分旧版账号管理的外层平台卡与内层账号行 CSS，修复紧凑视口折行。

## 审查边界

本审查基于 macOS 主机的 FastAPI + Vite 当前渲染截图；未完成打包 Tauri 黑盒验收，也未从截图推断完整无障碍合规。键盘焦点、屏幕阅读器语义和真实窗口缩放仍需单独测试。
