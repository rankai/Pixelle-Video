# React 短视频生产局部体验优化与发布助手方案

## Summary

本方案用于优化 React 桌面版短视频生产界面，访问入口为 `http://127.0.0.1:5173/`。本轮只改 React/Tauri 桌面端，不改 Streamlit 旧页面。

本轮不做大 UI 重构，不采用 A2/A3 新布局。保留当前口播生产主体验，在现有版本上做局部产品打磨、稳定性优化、AI 文案能力优化和桌面端发布助手能力。

核心判断标准：一个火锅店老板能沿用当前熟悉的 1-6 步流程，更清楚地获得操作反馈、更稳定地生成文案和成片，并能通过桌面端发布助手把视频快速准备到抖音发布页。

## Scope

### In Scope

- React 桌面版局部 UI 打磨和组件化。
- 左侧 `口播生产` 改为 `短视频生产`。
- 保留当前 1-6 步主流程和主界面结构。
- 发布页局部整理为更清晰的发布素材包。
- 状态提示、loading/error/success 展示统一。
- AI 改写/优化文案提示词升级。
- 桌面端浏览器自动化发布助手，首版优先抖音。
- 浏览器运行时可插拔：默认 Playwright，预留 CloakBrowser。
- `desktop/src/App.tsx` 逐步拆分，降低维护成本。
- 保持现有 FastAPI 接口和 IP 口播核心生产链路不变。

### Out of Scope

- Streamlit 旧页面。
- 大范围 UI 布局重构。
- A2/A3 三栏或向导式新布局。
- 改视频目标前台命名。
- 官方平台 API 发布优先方案。
- 新增复杂运营系统、CRM、团队权限。
- 重做音视频生成后端。
- 改变当前“整段配音 + 整段数字人 + 第 5 步画面覆盖”的生产策略。
- 绕过平台登录、验证码、审核或风控。

## Target Users

主要用户：

- 本地生活门店老板。
- 中小企业主。
- 门店运营人员。
- 本地生活代运营团队。

典型场景：

- 火锅店老板要发团购套餐口播。
- 洗衣店老板要发洗衣避坑干货。
- 美容院店长要发项目介绍。
- 民宿主理人要发房型/周边推荐。
- 代运营团队要帮多个门店批量生产视频。

用户特点：

- 忙于经营，不愿学习复杂软件。
- 不理解 workflow、ComfyUI、RunningHub 等技术概念。
- 需要直接可用的视频、标题、描述、标签和封面。
- 更关心“能不能发、发什么、哪里下载”，而不是参数细节。

## Product Principles

### 1. 业务目标优先

页面首先让用户回答“这条视频要干什么”，而不是先问技术参数。

推荐目标：

- 自由创作。
- 团购转化。
- 门店探店。
- 新品推荐。
- 老板人设。
- 客户案例。

业务目标影响：

- 文案结构。
- 画面规划建议。
- 默认模板。
- 发布平台建议。
- 标签和评论引导。

### 2. 默认路径短

默认路径必须清晰：

```text
选目标和素材 -> 确认文案 -> 生成配音 -> 生成数字人 -> 一键成片 -> 下载发布素材
```

高级参数不默认展开。老板用户只看到必要输入。

### 3. 每一步一个主动作

每一步只有一个最重要按钮。

示例：

- 第 1 步：`提取视频口播文案` / `生成口播文案`。
- 第 2 步：`AI 改写/优化文案`。
- 第 3 步：`生成配音`。
- 第 4 步：`生成出镜视频`。
- 第 5 步：`一键成片`。
- 第 6 步：`下载视频` / `复制发布素材`。

已完成步骤返回时：

- 主按钮：`使用当前结果继续`。
- 次按钮：`重新生成`。
- 不再显示容易误解的“只执行本步骤”。

### 4. 状态只显示一次

错误、loading、成功结果只在当前步骤动作区显示一次。

禁止：

- 顶部、主控台、底部同时出现同一错误。
- 技术错误直接暴露给业务用户。
- 长任务 loading 只藏在顶部按钮里。

### 5. 资产长期复用

音色、形象、模板、视频素材、品牌包必须独立维护。

工作台中允许快速添加素材，但完整管理仍在资产中心。

### 6. 发布结果产品化

第 6 步不是普通表单，而是“发布素材包交付台”。

用户最终应拿到：

- 最终视频。
- 标题。
- 描述。
- 标签。
- 评论引导。
- 封面。
- 平台建议。

## Information Architecture

### Left Navigation

建议导航：

```text
首页
短视频生产
素材资产
任务中心
系统配置
诊断工具
```

命名原则：

- 使用业务词，不使用技术词。
- “口播生产”升级为“短视频生产”，因为它比“视频生产”更贴近抖音、快手、小红书语境，也比“口播生产”覆盖探店、门店活动、产品种草等更多本地生活视频形态。
- 配置和诊断分开，避免普通用户误入诊断页。

### Home

首页定位：生产入口和系统就绪检查。

建议模块：

1. 主视觉区：
   - 产品名：`老板 IP 口播平台` 或 `门店短视频生产台`。
   - 副文案：`为本地生活门店快速生成可发布的口播和探店短视频。`
   - 主按钮：`新建视频`。

2. 轻量生产建议：
   - 本轮不做复杂选题助手。
   - 不新增前台视频目标命名体系。
   - 只显示当前流程可用的轻量入口，例如继续未完成任务或新建一条视频。
   - 点击后进入现有第 1 步，由用户在当前目标体系里选择本条视频目标。

3. 生产就绪检查：
   - LLM 配置。
   - 云端算力配置。
   - 音色是否有。
   - 数字人形象是否有。
   - 模板是否可用。

4. 当前任务：
   - 正在执行的任务。
   - 失败原因。
   - 继续生产入口。

5. 最近结果：
   - 最近生成的视频。
   - 下载/继续编辑入口。

首页不放复杂运营图表。首版只放和“今天能不能生产视频”“今天可以先做什么视频”直接相关的信息。

## Production Workflow Design

### Step 1：目标与素材

结构：

```text
顶部：本条视频目标
中部：素材来源
底部：当前来源输入区
底部动作栏：主按钮
```

本条视频目标：

- 轻量横向卡片或紧凑选择条。
- 默认 `自由创作`。
- 选择目标后显示一行“本目标会影响文案结构、画面建议、发布平台”。

本轮不改视频目标前台命名。视频目标相关优化只做文案说明和提示词效果，不改用户已经熟悉的目标名称。

素材来源：

- 视频提取。
- 粘贴脚本。
- 行业+人设。
- IP 学习。

行业+人设只问业务素材，不再问视频类型/文案类型：

- 行业/门店类型与人设身份。
- 产品/服务/活动与核心卖点。
- 适合什么客户。
- 优惠/预约/到店提示。
- 补充信息。

### Step 2：文案确认

目标：用户必须能检查最终口播稿。

规则：

- 第 1 步完成可自动跳第 2 步。
- 第 2 步 AI 改写后不能自动跳第 3 步。
- 改写后停留在第 2 步，让用户检查。
- 文案必须按自然语义分段，方便第 5 步画面规划。

UI：

- 主体为最终口播文案编辑器。
- 右侧或下方显示：
  - 字数。
  - 段落数。
  - 当前视频目标。
  - 改写设置折叠区。

### Step 3：配音制作

目标：用户能选音色、生成配音、试听。

规则：

- 生成后不自动跳第 4 步。
- 必须显示音频预览。
- 已生成状态下主按钮为 `使用当前配音继续`。
- 次按钮为 `重新生成配音`。

UI：

- 音色选择卡片。
- 工作流/高级参数折叠。
- 生成结果音频播放器。
- 失败时显示业务化错误。

### Step 4：数字人出镜

目标：用户能选择形象、生成数字人视频、预览。

规则：

- 生成后不自动跳第 5 步。
- 必须显示视频预览。
- 工作流自动过滤支持的形象类型。
- 不支持图片的工作流不显示图片形象。
- 不支持视频的工作流不显示视频形象。

UI：

- 工作流选择。
- 兼容形象卡片。
- 快速添加形象入口。
- 视频预览区。

### Step 5：一键成片

目标：把数字人视频、字幕、模板、画面覆盖、BGM 合成最终视频。

推荐结构：

1. 画面模板。
2. 画面规划。
3. 高级成片设置。
4. 成片动作区。

画面规划需要更业务化：

- 显示“哪些段落保持老板出镜”。
- 显示“哪些段落用门店/产品视频覆盖”。
- 支持勾选连续段落成组。
- 视频素材多时用弹窗选择，带搜索。

合成规则：

- 覆盖视频不足时必须循环或补齐，不得导致最终视频变短。
- BGM 长于视频时，最终长度以主视频/配音为准。
- 模板必须真正影响标题、字幕位置和样式。

### Step 6：发布素材包

目标：像交付一个完整素材包。

布局：

```text
左侧：视频预览和下载
右侧：发布素材包
底部：平台建议和复制动作
```

发布素材包包括：

- 标题。
- 描述。
- 标签。
- 评论引导。
- 封面。
- 口播文案。

交付清单：

```text
视频      已准备 / 缺失
封面      已准备 / 缺失
标题      已准备 / 缺失
描述      已准备 / 缺失
标签      已准备 / 缺失
口播文案  已准备 / 缺失
```

清单用途：

- 给用户明确完成感。
- 快速暴露缺失项。
- 避免用户不知道“现在能不能发”。
- 缺失项旁边提供对应修复入口，例如 `重新生成标题`、`生成封面`。

按钮：

- 下载最终视频。
- 复制标题。
- 复制描述。
- 复制标签。
- 复制整套发布素材。
- 发布到抖音。

## AI Copywriting Optimization

目标：优化第 2 步 `AI 改写/优化文案`，让输出更像本地生活门店老板真实口播，而不是泛营销文案。

必须保留：

- 分段结构。
- 最终文案可编辑。
- 视频目标对文案结构的影响。
- 不自动跳过用户确认。

提示词优化方向：

1. 老板视角：
   - 使用第一人称或门店经营者视角。
   - 减少空泛夸张表达。
   - 像老板对镜头自然讲，而不是广告旁白。

2. 本地生活语境：
   - 团购：突出优惠、套餐内容、适合人群、到店/下单引导。
   - 探店：突出场景、环境、体验细节、到店理由。
   - 新品：突出新品亮点、解决的问题、适合场景。
   - 老板人设：突出经验、避坑、专业判断。
   - 客户案例：突出前后变化、真实过程、结果可信度。

3. 画面规划友好：
   - 按自然语义分成 3-6 段。
   - 每段适合对应一个画面或覆盖组。
   - 不输出小标题、编号、markdown。

4. 发布友好：
   - 结尾给明确行动引导。
   - 不违规承诺效果。
   - 不使用绝对化、虚假夸张说法。

验收标准：

- 改写后仍保留换行分段。
- 文案读起来像真人口播。
- 文案里能体现当前视频目标。
- 第 5 步画面规划能基于段落继续使用。

## Browser Publishing Assistant

本轮自动化发布能力采用桌面端浏览器自动化发布助手，不优先采用官方 API 发布。

### Product Positioning

名称：`发布助手`。

定位：

```text
帮助用户把已生成的视频和发布素材自动准备到平台发布页，默认停在最终发布确认前。
```

首版平台：

```text
抖音
```

后续平台：

```text
快手
视频号
小红书
B站
```

### User Flow

```text
第 6 步生成发布素材包
  ↓
点击「发布到抖音」
  ↓
打开发布助手浏览器窗口
  ↓
用户首次登录抖音创作者后台
  ↓
系统复用登录态
  ↓
自动上传视频
  ↓
自动填写标题、描述、标签
  ↓
如有封面，尝试上传封面
  ↓
停在最终发布确认前
  ↓
用户确认后手动点击发布
```

### Boundaries

必须遵守：

- 不绕过登录。
- 不处理或绕过验证码。
- 不绕过平台审核。
- 不默认点击最终发布。
- 不做批量轰炸。
- 用户拥有最终确认权。
- 平台页面变更时停止并提示用户，而不是盲目点击。

### Technical Architecture

发布架构拆成两层：

```text
PublisherEngine
  ├─ BrowserRuntime
  └─ PlatformPublisher
```

浏览器运行时：

```text
BrowserRuntime
  ├─ PlaywrightRuntime      默认
  └─ CloakBrowserRuntime    可选
```

平台适配器：

```text
PlatformPublisher
  ├─ DouyinPublisher
  ├─ KuaishouPublisher
  ├─ WeChatChannelsPublisher
  ├─ XiaohongshuPublisher
  └─ BilibiliPublisher
```

统一发布输入：

```ts
type PublishPackage = {
  sessionId: string;
  videoPath: string;
  coverPath?: string;
  title: string;
  description: string;
  hashtags: string[];
  platform: "douyin" | "kuaishou" | "wechat_channels" | "xiaohongshu" | "bilibili";
};
```

统一发布结果：

```ts
type PublishResult = {
  status: "login_required" | "uploading" | "draft_ready" | "failed" | "cancelled";
  platform: string;
  message: string;
  screenshotPath?: string;
};
```

持久化目录：

```text
data/publish_browser/douyin/
```

任务记录：

```text
平台：抖音
任务：发布助手
状态：等待用户确认 / 已准备草稿 / 失败
时间
失败原因
```

### CloakBrowser Strategy

CloakBrowser 可以作为可选浏览器运行时纳入架构，但不作为首版唯一依赖。

使用原则：

- 默认运行时仍为 Playwright Chromium。
- CloakBrowser 作为 `BrowserRuntime` 的可选实现。
- 平台发布器不直接依赖 CloakBrowser。
- 不把产品定位成规避平台检测或风控的工具。
- 只在用户已登录、用户确认、平台规则允许的范围内做发布辅助。

引入 CloakBrowser 前需要验证：

- license 是否适合项目。
- macOS / Windows 打包体积。
- 二进制分发方式。
- Playwright API 兼容程度。
- 抖音、快手、视频号发布后台的稳定性。

### Web vs Desktop Boundary

纯 Web 形态适合：

- 下载视频。
- 复制标题/描述/标签。
- 打开平台后台链接。
- 官方 API 授权发布。

纯 Web 形态不适合：

- 稳定控制抖音/快手/视频号网页后台。
- 读取或操作第三方页面 DOM。
- 使用用户本机登录态。

桌面端适合：

- 启动受控浏览器。
- 保存平台登录态。
- 自动上传本地视频文件。
- 自动填写发布素材。
- 扩展多平台发布助手。

## Visual Direction

默认主题：石墨蓝专业版。

色彩：

- 背景：浅灰蓝。
- 主色：石墨蓝。
- 强调色：低饱和金橙，用于转化、提醒、主 CTA。
- 成功：绿色。
- 警告：琥珀色。
- 错误：红色。

布局：

- 深色侧栏建立产品识别。
- 内容区使用浅背景和白色面板。
- 卡片半径保持克制，建议 8-12px。
- 不使用大量渐变、大面积装饰图。
- 状态和结果比装饰更重要。

字体：

- 中文优先系统字体。
- 标题字重更明确。
- 表单说明小而清楚。
- 不用过大的营销型标题塞进工作区。

## Component Architecture

当前问题：

- `desktop/src/App.tsx` 超过 4200 行。
- `desktop/src/styles.css` 超过 2500 行。
- 页面、流程状态、资产管理、弹窗、样式逻辑混杂。

建议拆分：

```text
desktop/src/pages/
  HomePage.tsx
  VideoProductionPage.tsx
  AssetsPage.tsx
  TasksPage.tsx
  ConfigPage.tsx
  DiagnosticsPage.tsx

desktop/src/features/ip-production/
  ProductionController.tsx
  ProductionStepBar.tsx
  ProductionActionBar.tsx
  SourceStep.tsx
  ScriptStep.tsx
  VoiceStep.tsx
  PortraitStep.tsx
  ComposerStep.tsx
  PublishStep.tsx

desktop/src/features/assets/
  AssetCenter.tsx
  VoiceLibrary.tsx
  PortraitLibrary.tsx
  TemplateLibrary.tsx
  VideoAssetLibrary.tsx
  BrandKitLibrary.tsx

desktop/src/components/
  AppShell.tsx
  StatusNotice.tsx
  AssetCard.tsx
  ResultPreview.tsx
  EmptyState.tsx
  ConfirmDialog.tsx

desktop/src/styles/
  tokens.css
  layout.css
  components.css
  production.css
  assets.css
```

拆分原则：

- 先移动代码，不改行为。
- 每次拆一个区域。
- 每次拆分后跑 `npm run build`。
- 业务函数尽量从组件中抽离，避免 UI 组件承担流程编排。

## Phased Delivery

### Phase 1：局部体验和稳定性优化

目标：保留当前主界面结构，只修影响日常使用的反馈和稳定性问题。

工作：

- 左侧 `口播生产` 改为 `短视频生产`。
- 统一 loading/error/success 展示。
- 每一步只保留一个主要状态提示。
- 错误不在顶部、主控台、步骤底部重复出现。
- 已完成步骤返回时，主按钮为 `使用当前结果继续`，次按钮为 `重新生成`。
- 统一资产卡片按钮、预览、试听的基础样式。
- 不改变当前 1-6 步布局。

验收：

- `npm run build` 通过。
- 第 1-6 步仍可进入。
- 任一步失败只显示一个错误提示。
- 长任务 loading 在当前动作区可见。
- 左侧导航显示 `短视频生产`。

### Phase 2：AI 改写提示词优化

目标：让 `AI 改写/优化文案` 更适合本地生活门店老板口播。

工作：

- 优化 `build_rewrite_prompt()`。
- 强制自然分段。
- 强化本地生活场景和老板口吻。
- 保留视频目标对结构的影响。
- 增加 prompt 单测。

验收：

- 改写后仍保留 3-6 段自然分段。
- 文案不是广告腔、不是小标题堆叠。
- 当前视频目标能影响结构。
- 第 5 步画面规划仍能读取段落。

### Phase 3：发布素材包局部整理

目标：第 6 步结果交付清晰。

工作：

- 视频预览和下载区。
- 标题/描述/标签/评论引导卡片。
- 发布交付清单。
- 一键复制整套素材。
- 平台建议区。

验收：

- 最终视频可预览。
- 下载按钮有效。
- 标题/描述/标签可复制。
- 用户能一眼看到视频、封面、标题、描述、标签、口播文案是否准备好。
- 发布页不再是散乱表单。

### Phase 4：组件化重构，不改 UI 行为

目标：降低 `App.tsx` 和 `styles.css` 后续维护成本。

工作：

- 拆 `AppShell`。
- 拆页面组件。
- 拆口播 1-6 步组件。
- 拆资产卡片与通用状态提示组件。
- 保持视觉和行为基本不变。

验收：

- `npm run build` 通过。
- 页面可正常切换。
- 口播 1-6 步能进入。
- 无明显 UI 回归。

### Phase 5：抖音发布助手 v1

目标：在桌面端通过浏览器自动化把发布素材准备到抖音发布页。

工作：

- 新增发布模型和发布服务。
- 新增 `BrowserRuntime` 抽象。
- 新增 `PlaywrightRuntime` 默认实现。
- 预留 `CloakBrowserRuntime` 可选实现。
- 新增 `DouyinPublisher`。
- 使用持久化用户数据目录 `data/publish_browser/douyin/`。
- 第 6 步增加 `发布到抖音`。
- 任务中心记录发布助手状态。

验收：

- 首次使用时能打开抖音发布助手窗口。
- 未登录时提示用户登录，不绕过登录或验证码。
- 登录后能上传最终视频。
- 能填写标题、描述、标签。
- 默认停在最终发布确认前。
- 失败时保留截图或可读错误信息。

### Phase 6：多平台发布扩展

目标：复用发布助手架构扩展更多平台。

平台顺序：

1. 快手。
2. 视频号。
3. 小红书。
4. B站。

验收：

- 新平台通过 `PlatformPublisher` 适配器接入。
- 不复制发布引擎逻辑。
- 各平台登录/验证码/页面变化有明确错误提示。

### Phase 7：资产中心统一

目标：长期复用素材更清楚。

工作：

- 统一资产卡片。
- 音色试听。
- 形象预览。
- 视频素材搜索。
- 模板预览。
- 快速添加弹窗风格统一。

验收：

- 每类资产的卡片高度、按钮、预览一致。
- 视频素材多时可搜索。
- 流程内快速添加体验和资产中心一致。

## Risks

### Product Value Review

按照本方案实施后，预期是提升产品力和使用体验，而不是倒退。判断依据如下。

产品力提升点：

1. 当前体验稳定性提升：
   - 保留用户已经熟悉的主流程。
   - loading/error/success 更清楚。
   - 不通过大布局重构增加新的学习成本。

2. AI 文案质量提升：
   - 文案更接近本地生活老板真实口播。
   - 分段更稳定，便于画面规划。
   - 减少 AI 营销腔和空泛表达。

3. 发布素材包增强完成感：
   - 交付清单让用户知道是否可以发布。
   - 视频、封面、标题、描述、标签、口播文案都可见。
   - 结果从“字段堆叠”升级为“可交付素材包”。

4. 桌面端发布助手形成差异化：
   - 自动上传和填写发布内容，减少老板手动搬运。
   - 用户保留最终确认权。
   - 可扩展到快手、视频号、小红书等平台。

5. 流程状态更稳定：
   - loading/error 只显示一次。
   - 每一步一个主动作。
   - 第 2/3/4 步完成后保留确认机会，符合用户检查文案、试听配音、预览出镜视频的真实习惯。

6. 工程可维护性提升：
   - 拆分 `App.tsx` 和样式文件后，后续迭代成本下降。
   - 页面、流程步骤、资产卡片、发布素材包能独立演进。

可能倒退的风险：

1. 如果把局部优化扩展成大布局重构，会造成学习成本上升。
   - 规避：保留当前主界面结构，只做局部打磨。

2. 如果发布助手变成自动绕风控工具，会带来合规和平台风险。
   - 规避：默认停在最终发布确认前，不绕过登录、验证码或审核。

3. 如果 CloakBrowser 绑定过深，会增加打包和维护风险。
   - 规避：CloakBrowser 只作为可选 runtime，默认使用 Playwright。

4. 如果发布页只做视觉卡片但下载/复制不可用，会造成产品倒退。
   - 规避：发布页验收必须包含预览、下载、复制和交付清单状态。

5. 如果 AI 改写强行套结构，会让口播不自然。
   - 规避：提示词要求自然分段，不输出标题、编号或 markdown。

结论：本方案的价值增益大于风险，但前提是严格按阶段执行。不能先大面积改视觉，也不能把发布助手做成绕平台规则的黑盒。成功标准不是“更好看”，而是火锅店老板能更稳定地生成视频、更容易拿到可发布素材，并能减少手动发布搬运。

### Risk 1：一次性重构过大

缓解：

- 先拆分组件，再改布局。
- 每个阶段单独提交。
- 每个阶段都能回滚。

### Risk 2：视觉变好但流程变复杂

缓解：

- 每一步只保留一个主动作。
- 高级参数折叠。
- 用火锅店老板场景做验收。

### Risk 3：发布页展示好看但结果不可用

缓解：

- 下载、复制、预览必须作为验收项。
- 不允许只做静态 UI。

### Risk 4：组件拆分后状态混乱

缓解：

- 保留现有 session API。
- 先通过 props 传递状态，不提前引入复杂状态库。
- 后续如有必要再引入轻量 store。

## Non-Goals

- 不做智能运营系统。
- 不做复杂数据看板。
- 不做多账号团队协作。
- 不接真实发布平台。
- 不把素材上传变成复杂识别流程。
- 不在本轮重构 Streamlit。

## Definition of Done

本轮 UI 重构完成时必须满足：

- React 页面结构明显更清晰。
- 口播工作台 1-6 步主流程完整可跑。
- 发布素材包清晰可用。
- 资产选择和预览统一。
- 普通老板用户不需要理解技术参数即可完成视频。
- `desktop/src/App.tsx` 明显瘦身。
- `npm run build` 通过。
- 关键后端测试通过。
